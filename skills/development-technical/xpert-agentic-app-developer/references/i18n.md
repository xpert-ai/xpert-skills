# Plugin i18n Implementation

Use this reference when implementing, reviewing, or migrating user-visible text in an Xpert Workbench view, extension view, remote component, generated artifact, or view action.

## Architecture

Expose one plugin-owned i18n facade per UI surface. Components depend on that facade, not directly on a translation engine or the host bridge. Prefer an established plural- and interpolation-capable engine such as `i18next` with `react-i18next`, or the host platform equivalent. A lightweight custom resolver is acceptable only when it provides typed keys, parameter interpolation, plural rules, locale fallback, and catalog-completeness validation.

The facade should expose a small stable API:

```ts
interface PluginI18n {
  locale: SupportedLocale
  t(key: MessageKey, values?: MessageValues): string
  formatDate(value: Date | string, options?: Intl.DateTimeFormatOptions): string
  formatNumber(value: number, options?: Intl.NumberFormatOptions): string
  formatCurrency(value: number, currency: string): string
  formatPercent(value: number): string
}
```

Organize catalogs by view or domain namespace instead of one global file:

```text
remote-components/
  i18n/
    core.ts
    react.tsx
    formatters.ts
    platform-adapter.ts
    common/
      en-US.ts
      zh-Hans.ts
  contract-review/src/i18n/
    en-US.ts
    zh-Hans.ts
```

Keep shared actions, statuses, accessibility labels, and validation messages in `common`. Keep domain-specific copy beside its remote component so independent iframe bundles do not carry every plugin message.

## Locale Boundary

Treat the locale in the host iframe `init` message as authoritative. Normalize it once before rendering and publish the normalized value through the i18n facade. Do not read `navigator.language` after the host locale is available and do not repeat normalization inside components.

Use BCP 47 runtime locale tags such as `en-US`, `zh-Hans`, `zh-Hant`, and `ja-JP`. Keep an explicit registry and fallback chain:

```ts
const SUPPORTED_LOCALES = ['en-US', 'zh-Hans', 'zh-Hant'] as const
const DEFAULT_LOCALE = 'en-US'
```

Normalize underscore platform aliases at the boundary, for example `en_US -> en-US` and `zh_Hans -> zh-Hans`. Map known regional aliases deliberately, such as `zh-CN -> zh-Hans` and `zh-TW -> zh-Hant`. Never use `startsWith('zh')` to map every Chinese locale to Simplified Chinese.

Resolve locale in this order:

1. Exact supported BCP 47 tag.
2. Explicit language/script or regional alias.
3. Configured default for that base language.
4. `en-US` for a reusable plugin.

Set `document.documentElement.lang` to the resolved locale and derive `dir` from locale metadata when RTL languages are supported.

## Catalog Design and Typing

Use semantic keys that survive copy edits:

```text
quotation.empty.lockCostingFirst
validation.targetMargin.max
status.technicalLocked
actions.confirm
```

Do not use English sentences as keys and do not name keys after component line numbers or layout positions.

Use the default-locale catalog as the message schema. Type other locale catalogs against the same deep key shape so builds fail when a key is missing or misplaced. Type translation calls so unknown keys and invalid parameter names fail compilation where the selected engine supports it.

Catalog leaves contain complete phrases. Do not assemble translated sentences by concatenating fragments because word order changes across languages.

## React Usage

Use a provider or equivalent hook at the remote-component root:

```tsx
function QuotationView() {
  const { t } = usePluginI18n('quickQuotation')
  return <EmptyState>{t('quotation.empty.lockCostingFirst')}</EmptyState>
}
```

Do not write locale branches in JSX or component helpers:

```tsx
// Avoid
locale === 'zh-Hans' ? '已锁定' : 'Locked'
```

All user-visible labels, tooltips, placeholders, empty states, validation errors, confirmations, toast messages, table headers, filenames, `aria-label` values, and image alternative text go through the translator. Domain data such as customer names, material descriptions, evidence text, and audit payloads remain unchanged unless the data contract explicitly provides localized variants.

Use controlled rich-text component placeholders for translated copy that contains links or emphasis. Never store executable HTML in catalogs.

## Formatting and Dynamic Messages

Use interpolation instead of string concatenation:

```ts
t('results.summary', { matched, total })
```

Use the translation engine's plural rules backed by `Intl.PluralRules`; do not assume that singular/plural is an English-only two-way branch. Use `Intl.DateTimeFormat`, `Intl.NumberFormat`, currency, and percent formatters from the facade rather than locale ternaries.

Keep stable domain values as language-neutral codes and translate only their display labels:

```ts
t(`status.${status}`)
```

Do not generate English labels with `replaceAll('_', ' ')`; every user-visible enum requires an explicit display mapping and an unknown-value fallback.

## Platform and Backend Contracts

Keep platform metadata localized objects at the platform boundary:

```ts
title: { en_US: 'Contract Review', zh_Hans: '合同审核' }
```

Centralize conversion between platform keys such as `en_US` / `zh_Hans` and runtime keys such as `en-US` / `zh-Hans` in `platform-adapter.ts`. Business components must not index localized platform objects themselves.

Prefer backend and Agent middleware results with stable codes and parameters:

```json
{
  "errorCode": "price_list_import_failed",
  "messageKey": "errors.priceList.importFailed",
  "params": { "row": 12 }
}
```

Keep `errorCode`, status, and structured recovery data even when a localized display message is also returned. Support legacy localized objects through the platform adapter with an explicit fallback order; do not spread `value[locale === ...]` branches through UI code.

Pass a normalized locale when generating user-facing Excel, PDF, Word, email, or export filenames. Persist language-neutral audit facts alongside any localized artifact text.

## Third-Party Components

Centralize locale mapping for editors and component libraries:

```ts
const UNIVER_LOCALE_BY_APP_LOCALE: Record<SupportedLocale, LocaleType> = { /* ... */ }
```

Keep unsupported third-party locale fallback explicit. A third-party locale map is an allowed locale branch boundary; ordinary view components are not.

## Validation and Migration

Add automated checks that:

- Compare every locale catalog with the default catalog for missing and extra keys.
- Type-check translation keys and required interpolation parameters.
- Reject inline locale-to-text branches outside i18n, platform-adapter, artifact-generation, and third-party locale adapter files.
- Render each remote view in every supported locale and verify important accessibility labels.
- Test interpolation, pluralization, locale fallback, date/number/currency formatting, and unknown enum values.
- Exercise an expanded pseudo-locale such as `en-XA` to expose clipping and fixed-width assumptions.
- Rebuild generated iframe assets and check that they are not stale.

Migrate incrementally:

1. Inventory user-visible literals, local `COPY` / `MESSAGES` dictionaries, status maps, and locale ternaries.
2. Add the shared facade, supported-locale registry, and platform adapter without changing persisted codes.
3. Move one remote view into a namespace and replace component locale branches with typed `t(...)` calls.
4. Centralize enum labels and `Intl` formatters.
5. Add catalog parity and no-inline-locale-branch checks before migrating the next view.
6. Use a pseudo-locale or a real third locale to prove that adding a language changes catalogs and registration, not business components.
